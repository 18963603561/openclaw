# 语音唤醒与按住说话

## 模式

- **唤醒词模式**（默认）：始终开启的 Speech 识别器会等待触发词（`swabbleTriggerWords`）。匹配后，它会开始采集，显示带有部分文本的浮层，并在静音后自动发送。
- **按住说话（按住右 Option 键）**：按住右侧 Option 键可立即开始采集——无需触发词。按住期间会显示浮层；松开后会完成并转发，期间会有一个短暂延迟，方便你微调文本。

## 运行时行为（唤醒词）

- Speech 识别器位于 `VoiceWakeRuntime` 中。
- 只有当唤醒词与下一个词之间存在**明显停顿**时，触发才会生效（约 `0.55s` 间隔）。浮层 / 提示音可以在停顿时就开始，即使命令尚未开始。
- 静音窗口：当语音持续流动时为 `2.0s`，如果只听到了触发词则为 `5.0s`。
- 硬停止：`120s`，用于防止会话失控持续。
- 会话之间的防抖间隔：`350ms`。
- 浮层通过 `VoiceWakeOverlayController` 驱动，使用已确认 / 临时着色。
- 发送后，识别器会干净地重新启动，以监听下一个触发词。

## 生命周期不变量

- 如果已启用语音唤醒且权限已授予，则唤醒词识别器应处于监听状态（显式按住说话采集期间除外）。
- 浮层可见性（包括通过 X 按钮手动关闭）绝不能阻止识别器恢复。

## 浮层卡住的失败模式（之前）

此前，如果浮层卡在可见状态而你手动将其关闭，语音唤醒可能看起来像“失效”了一样，因为运行时的重启尝试可能会被浮层可见性阻塞，且之后不会再安排新的重启。

加固措施：

- 唤醒运行时的重启不再受浮层可见性阻塞。
- 浮层关闭完成后，会通过 `VoiceSessionCoordinator` 触发 `VoiceWakeRuntime.refresh(...)`，因此手动点击 X 关闭后始终会恢复监听。

## 按住说话细节

- 热键检测使用针对**右侧 Option** 的全局 `.flagsChanged` 监视器（`keyCode 61` + `.option`）。我们只观察事件（不会拦截）。
- 采集流水线位于 `VoicePushToTalk` 中：立即启动 Speech，将部分结果流式传输到浮层，并在松开时调用 `VoiceWakeForwarder`。
- 当按住说话开始时，我们会暂停唤醒词运行时，以避免音频 tap 相互冲突；它会在松开后自动重启。
- 权限：需要麦克风 + Speech；要看到事件则需要辅助功能 / 输入监控批准。
- 外接键盘：某些键盘可能不会按预期暴露右侧 Option——如果用户反馈漏检，请提供备用快捷键。

## 面向用户的设置

- **语音唤醒** 开关：启用唤醒词运行时。
- **按住 Cmd+Fn 说话**：启用按住说话监视器。在 macOS < 26 上禁用。
- 语言与麦克风选择器、实时电平表、触发词表、测试器（仅本地；不会转发）。
- 麦克风选择器会在设备断开连接时保留上次选择，显示断开提示，并在设备恢复前临时回退到系统默认设备。
- **声音**：在检测到触发词和发送时播放提示音；默认使用 macOS 的 “Glass” 系统声音。你可以为每个事件选择任意可由 `NSSound` 加载的文件（例如 MP3 / WAV / AIFF），也可以选择**无声音**。

## 转发行为

- 启用语音唤醒后，转录内容会被转发到当前激活的 Gateway 网关 / 智能体（与 mac 应用其余部分使用的本地或远程模式一致）。
- 回复会发送到**上次使用的主提供商**（WhatsApp / Telegram / Discord / WebChat）。如果投递失败，错误会被记录，且此次运行仍可通过 WebChat / 会话日志查看。

## 转发载荷

- `VoiceWakeForwarder.prefixedTranscript(_:)` 会在发送前添加机器提示前缀。该逻辑由唤醒词路径和按住说话路径共享。

## 快速验证

- 打开按住说话开关，按住 Cmd+Fn，讲话，然后松开：浮层应显示部分结果，然后发送。
- 按住期间，菜单栏耳朵图标应保持放大状态（使用 `triggerVoiceEars(ttl:nil)`）；松开后会恢复。
